Le Principe DRY

Définition — Andy Hunt & Dave Thomas, The Pragmatic Programmer (1999)

"Every piece of knowledge must have a single, unambiguous, authoritative representation within a system."

En pratique : si vous modifiez une règle métier et que vous devez toucher plus d'un endroit dans le code, DRY est violé. Le copier-coller de logique est l'ennemi n°1 de la maintenabilité.

Symptômes de violation DRY

  • • Copier-coller d'une même formule dans plusieurs méthodes
  • • Modifier un taux de taxe oblige à toucher 3 fichiers
  • • Code identique dans plusieurs contrôleurs / services
  • • SonarQube signale des "Duplicated Lines"

Après refactoring DRY

  • • La règle métier vit à un seul endroit
  • • Changer le taux de taxe = modifier 1 ligne
  • • Les méthodes spécifiques délèguent à la logique commune
  • • SonarQube : Duplications = 0.0%

Un apprenti développeur a créé trois routes de paiement pour les guildes de la taverne. Le code fonctionne. Analysez-le attentivement avant de commencer à le modifier.

Réflexions pour votre rapport

  • 1. Que pensez-vous de la qualité de ce code ? Justifiez votre analyse.
  • 2. Imaginez que le taux de taxe ou la formule de calcul évolue. Qu'est-ce que cela implique concrètement dans ce code ? Est-ce satisfaisant ?
  • 3. Comment refactoreriez-vous ce code pour le rendre plus maintenable ? Décrivez et justifiez votre approche.
3

Mission

Tâches à réaliser

  1. 1

    Ouvrez le projet starter dans votre IDE

    Lancez l'application et vérifiez que les 3 routes /warrior, /mage, /rogue répondent correctement avant de modifier quoi que ce soit.

  2. 2

    Créez un service de calcul dédié

    Extrayez la logique commune dans une classe annotée @Service. Le contrôleur doit déléguer à ce service via injection de dépendances.

  3. 3

    Conservez les 3 routes et leurs spécificités

    Les routes /warrior, /mage, /rogue doivent continuer à exister et renvoyer les bons résultats. Seule la logique dupliquée doit migrer dans le service.

  4. 4

    Test de régression

    Vérifiez via Postman/cURL que les 3 routes renvoient les mêmes résultats qu'avant le refactoring. Le comportement ne doit pas changer.

  5. 5

    Vérifiez avec SonarQube

    Lancez l'analyse. La métrique Duplicated Lines (%) doit afficher 0.0%.

Contrainte DRY à respecter

La formule prix × quantité et le calcul de la taxe standard du Roi (5%) ne doivent apparaître qu'une seule fois dans tout le code source après votre refactoring.
